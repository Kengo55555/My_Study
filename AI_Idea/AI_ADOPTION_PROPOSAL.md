# AI駆動開発 導入提案書
## 組み込みソフトウェア開発チーム向け — 10の課題と推進ステップ

> **本資料の目的**
> 貴社が抱える設計開発現場の課題に対し、AIを段階的に導入することで「開発速度の向上」「品質の維持・向上」「ナレッジの組織資産化」を実現するための具体的な推進計画を提示する。

---

## はじめに：なぜ今、組み込み開発にAIが必要か

組み込みソフトウェア開発は、ハードウェアとの密結合・リアルタイム制約・安全規格への準拠など、他分野よりも要求水準が高い。一方、以下の環境変化により従来の開発手法だけでは対応が困難になりつつある。

```
【外部環境の変化】
製品サイクルの短縮 → 開発スピードへの要求が増大
ソフトウェア規模の拡大 → 人手だけでの品質担保が限界に
熟練技術者の高齢化 → ナレッジ消失リスクが高まっている
```

AIは「人間を置き換える技術」ではなく、**「熟練技術者の判断力を若手に伝え、開発プロセスを加速する基盤」** として位置付けることが重要である。

## 課題別：具体的な推進ステップ

### 課題1：AI駆動開発のスピード化

**背景・現状**

設計・開発の各フェーズで「調べる・書く・確認する」作業が個人に属人化しており、過去の事例が組織資産として蓄積されていない。同じ調査を複数人が繰り返すロスが生じている。

**解決後の姿**

AIが過去事例・設計ドキュメントを検索し、初稿を提示。人間は「正しいか判断する」役割に移行する。

**推進ステップ**

```
Step 1【1ヶ月目】
  過去の設計書・議事録・トラブル事例を棚卸し
  → どのドキュメントがAIに読み込ませる価値があるか整理

Step 2【2ヶ月目】
  RAGシステム（社内ナレッジAI）を試験構築
  → 過去ドキュメントを検索してAIが回答するPoC

Step 3【3〜4ヶ月目】
  特定チーム・特定業務での試験運用
  → 設計初稿生成・類似事例検索をAIで補助

Step 4【5〜6ヶ月目】
  効果測定・ルール整備 → 全チームへ横展開
```

**KPI（成果指標）**

| 指標 | 目標 |
|---|---|
| 設計初稿作成時間 | 現状比 30% 削減 |
| 過去事例の参照件数 | 月10件以上（現状ほぼゼロ） |

### 課題2：スキル育成・評価制度の整備

**背景・現状**

AIツールの習得が個人の自助努力に委ねられており、組織としてのAIリテラシーに大きなばらつきがある。「使える人」と「使えない人」の分断が生じ始めている。

**解決後の姿**

AI活用スキルを3段階（初級・中級・上級）で定義し、習得状況を可視化。評価制度に組み込む。

**スキルレベル定義（案）**

| レベル | 定義 | 具体的な能力の例 |
|---|---|---|
| Level 1（初級） | AIツールを日常的に使える | Copilotでコード補完・議事録生成ができる |
| Level 2（中級） | AIをプロセスに組み込める | CI/CDにAIレビューを設定できる。CLAUDE.mdを書ける |
| Level 3（上級） | AIを設計・活用できる | RAGシステムを構築できる。AIエージェントを作れる |

**推進ステップ**

```
Step 1【1ヶ月目】
  スキルレベルの定義をチームで合意（ベテランも参加させる）

Step 2【2ヶ月目】
  全員のAIリテラシー現状調査（匿名アンケート）

Step 3【3ヶ月目】
  Level 1 到達のためのハンズオン研修実施（月1回）

Step 4【4〜6ヶ月目】
  人事評価へのAIスキル項目の追加を検討・提案
```

### 課題3：開発ツールへのAI統合

**背景・現状**

コードを書きながら別画面でAIチャットを使う「分断した使い方」が主流。作業が中断されるため、生産性向上効果が限定的。

**解決後の姿**

IDEの中でAIがリアルタイムに補完・レビュー・テスト生成まで完結する。手を止めずに開発が進む。

**推進ステップ**

```
Step 1【1ヶ月目】
  VS Code + GitHub Copilot を全員がインストール・試用
  → 2週間の試用後に感想共有会

Step 2【2ヶ月目】
  CLAUDE.md（プロジェクトルール定義ファイル）を整備
  → ベテランが「AIへの指示書」を書く役を担当

Step 3【3ヶ月目】
  Claude Code（CLIエージェント）の試験導入
  → 複数ファイルの一括編集・テスト生成を体験

Step 4【4ヶ月目以降】
  使用ツールの標準化・ライセンスの整備
```

**CLAUDE.md の最小構成例**

```markdown
## コーディング規約
- MISRA-C 2012 準拠
- malloc/free 禁止
- 関数は50行以内、1関数1責務

## 必須ドキュメント
- 全関数にDoxygenコメントを付けること
- 設計変更時はADRフォルダに記録すること

## AIへの禁止事項
- 割り込みハンドラ内にOS API呼び出しを入れないこと
- NULLポインタチェックを省略しないこと
```

### 課題4：コード品質・レビューの効率化

**背景・現状**

コードレビューが特定のベテランに集中。レビュー待ちがボトルネックになり、開発速度が落ちている。ベテランの負担も限界に近い。

**解決後の姿**

AIが一次レビューを実施。明らかな問題はAIが指摘し、ベテランは「高難度の判断」にのみ集中する役割分担を実現。

**推進ステップ**

```
Step 1【1ヶ月目】
  ベテランに「よくある指摘パターン」をヒアリング
  → CLAUDE.md / レビューチェックリストに落とし込む

Step 2【2ヶ月目】
  AIレビューを1リポジトリで試験導入
  → PRを出すとAIがコメントを自動投稿

Step 3【3ヶ月目】
  AI指摘 vs ベテラン指摘の比較・精度評価
  → AIのルールを随時アップデート

Step 4【4ヶ月目以降】
  全リポジトリへ横展開
  → ベテランのレビュー工数を定量的に測定・改善
```

**期待効果**

| 指標 | 目標 |
|---|---|
| ベテランのレビュー時間 | 現状比 40% 削減 |
| レビュー指摘の一次対応時間 | 当日中（現状：数日待ち） |
| MISRA違反の後工程発覚件数 | 現状比 70% 削減 |

### 課題5：コーディング規約の自動遵守（MISRA-C等）

**背景・現状**

MISRA-C準拠チェックが静的解析ツールの手動実行に依存。コミット後・レビュー後に違反が発覚し、手戻りが頻発している。

**解決後の姿**

コーディング中・コミット前にリアルタイムで規約違反が検出される。手戻りがなくなり、レビューの質が向上する。

**推進ステップ**

```
Step 1【1ヶ月目】
  現在の規約違反の発生箇所・頻度を調査（過去3ヶ月分の静的解析ログを分析）

Step 2【2ヶ月目】
  CLAUDE.md に規約を全て記載し、AIがコード生成時に自動適用する設定

Step 3【3ヶ月目】
  git commit 前フック（pre-commit hook）に静的解析を自動実行する設定
  → 違反があればコミット自体をブロック

Step 4【4ヶ月目】
  CI/CDパイプラインとの統合（後述の課題7と連携）
```

**違反検出タイミングの比較**

```
現状：
コード作成 → コミット → レビュー → 違反発覚 → 修正 → 再レビュー

改善後：
コード作成 → [AIがリアルタイム警告] → コミット前に修正 → クリーンなPR
```

### 課題6：テスト自動化・カバレッジ向上

**背景・現状**

テスト設計・テストコード作成が手作業のため、テストカバレッジが低い状態で出荷せざるを得ない。テストの属人化も課題。

**解決後の姿**

AIが関数仕様からテストケースとテストコードを自動生成。カバレッジが向上し、テスト設計の属人性が解消される。

**推進ステップ**

```
Step 1【1ヶ月目】
  現在のテストカバレッジを計測・可視化（ベースライン確立）

Step 2【2ヶ月目】
  新規追加関数を対象にAIテスト生成を試験導入
  → 正常系・異常系・境界値のテストケースをAIが提案

Step 3【3ヶ月目】
  CUnit / Google Test 形式でのコード自動生成
  → エンジニアがレビュー・追記

Step 4【4〜6ヶ月目】
  CI/CDに組み込み、カバレッジを自動計測・レポート化
  → カバレッジ基準値（例：80%以上）を設けてゲートとして機能させる
```

### 課題7：CI/CDパイプラインの整備

**背景・現状**

ビルド・静的解析・テスト実行が手動。実行漏れや「自分のPCでは動く」問題が発生。手動実行の工数が開発者の負担になっている。

**解決後の姿**

GitHub Actions（またはGitLab CI）でpushのたびに全チェックが自動実行。「人間が忘れる」問題がなくなる。

**パイプライン構成（最終形）**

```
開発者が git push
       ↓
GitHub Actions 起動
       ↓
① クロスコンパイル（ターゲットMCU向けビルド）
       ↓
② 静的解析（MISRA-C 違反チェック）
       ↓
③ ユニットテスト自動実行
       ↓
④ テストカバレッジ計測・レポート生成
       ↓
⑤ AIコードレビューコメントをPRに自動投稿
       ↓
全通過 → レビュワー（人間）へ通知
```

**推進ステップ**

```
Step 1【1ヶ月目】
  GitHub / GitLabの利用状況確認。リポジトリ構成の整理

Step 2【2ヶ月目】
  最もシンプルなパイプライン（ビルドのみ）をまず構築・動作確認

Step 3【3ヶ月目】
  静的解析・ユニットテストをパイプラインに追加

Step 4【4ヶ月目以降】
  AIレビューの組み込み → 完全自動化パイプラインの完成
```

### 課題8：ドキュメント管理の自動化（SharePoint連携）

**背景・現状**

設計書・仕様書の作成・更新が後回しになりがち。SharePoint上に古い資料と新しい資料が混在し、「どれが正しいか」の確認コストが高い。

**解決後の姿**

コードと設計書が常に同期された状態を維持。AIがコードからドキュメントの初稿を生成し、SharePointに自動連携する。

**推進ステップ**

```
Step 1【1ヶ月目】
  SharePoint上のドキュメントを棚卸し
  → 最新版・廃止版の整理。命名規則の統一

Step 2【2ヶ月目】
  Doxygenコメントの自動生成をAIで運用開始
  → コードにコメントがあれば自動でHTML/PDFに変換

Step 3【3ヶ月目】
  設計変更時の仕様書初稿生成をAIで補助
  → 変更差分を入力するとAIが更新箇所のドラフトを生成

Step 4【4ヶ月目以降】
  CI/CDと連携してdocument更新を自動化
  → mergeのタイミングでSharePointの文書を自動更新
```

**ドキュメント階層の整理案**

```
SharePoint/
├── 現行仕様書/（AIとCI/CDが自動更新）
├── 設計判断記録（ADR）/（変更理由を蓄積）
├── 過去バージョン/（アーカイブ）
└── ナレッジベース/（AIの学習元となるドキュメント）
```

### 課題9：Copilot Studio活用（社内AI窓口の構築）

**背景・現状**

技術的な問い合わせが特定のベテランに集中。「○○さんに聞かないとわからない」という属人化が常態化。ベテランの退職リスクも高まっている。

**解決後の姿**

Microsoft Copilot Studioで「社内専門家AI」を構築。過去の設計書・トラブル事例・ノウハウを学習させ、誰でも24時間回答を得られる状態にする。

**構成イメージ**

```
社内ドキュメント（SharePoint・過去議事録・設計書・ADR）
       ↓
Copilot Studio が読み込み・インデックス化
       ↓
Teamsチャットから質問
「このレジスタ設定の意味は何ですか？」
「FreeRTOSでセマフォとミューテックスの使い分けは？」
       ↓
AIが関連ドキュメントを検索して回答（出典付き）
       ↓
不明な場合はベテランへエスカレーション
```

**推進ステップ**

```
Step 1【1ヶ月目】
  Copilot Studioのライセンス確認・PoC環境の準備

Step 2【2ヶ月目】
  最も問い合わせが多い「よくある質問」トップ20を整理
  → ドキュメント化してCopilot Studioに学習させる

Step 3【3ヶ月目】
  特定チームでTeams上のAIチャットボットを試験運用
  → 回答精度のフィードバックを収集

Step 4【4ヶ月目以降】
  全社展開 → ベテランへの問い合わせ件数を定量的に削減
```

### 課題10：コード生成・レガシーコード対応

**背景・現状**

コメントのない旧コードの解読に多大な時間を要している。担当者引き継ぎのたびに「コード読解」から始まるため、属人化が解消されない。

**解決後の姿**

AIがレガシーコードを解読・解説・ドキュメント化。新規コードはAIが初稿を生成し、人間が検証する分業体制が確立される。

**推進ステップ**

```
Step 1【1ヶ月目】
  最も属人化している・最も解読困難なファイルをリストアップ
  → 優先順位をつけて対象を絞る

Step 2【2ヶ月目】
  対象ファイルをAIに読み込ませて解説・コメント生成
  → 出力内容をベテランがレビューして修正（これがナレッジ移転になる）

Step 3【3ヶ月目】
  新規機能追加はAIが初稿を生成するフローを試験導入
  → レビューチェックリストに基づいてエンジニアが確認・修正

Step 4【4ヶ月目以降】
  レガシーコードの計画的なリファクタリングにAIを活用
  → 影響範囲調査・テスト生成・修正の一連をAIが支援
```

## 成功のための3原則

### 原則1：「ベテランがAIを使う」ではなく「AIがベテランを支える」
ベテランの承認・判断を最終ステップに置き続ける。AIはベテランを脅かすものではなく、雑務を引き受けてベテランの本来の仕事に集中させる存在として位置付ける。

### 原則2：小さく始めて、数字で語る
最初から全課題を一斉に動かさない。1〜2の課題でPoC → 効果測定 → 横展開のサイクルを回すことで、失敗リスクを最小化し「成功体験」を積み上げる。

### 原則3：AIのルールはチームが作る
CLAUDE.md・コーディング規約・レビューチェックリストは、AIに詳しいメンバーだけでなくベテランが主導して作成する。「自分たちがAIを制御している」という主体感を持たせることが、持続的な活用の鍵になる。
