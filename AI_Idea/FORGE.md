## 1. 設計支援

### アイデア：ジェネレーティブデザイン補助

**概要説明**

制約条件（重量・強度・コスト・材料）を入力すると、AIが数百〜数千の設計案を自動生成するアイデア。人間が思いつかない形状を提案し、設計者は「選ぶ人」になる。

**たとえ話**
> 従来の設計は「白紙から絵を描く」作業。ジェネレーティブデザインは「AIが描いた100枚のスケッチから最高のものを選ぶ」作業に変わる。

**活用フロー**

```
設計者が制約を入力
（重量 < 5kg、耐荷重 > 200N、材料：アルミ合金）
       ↓
AIが候補形状を大量生成
       ↓
FEA（有限要素解析）で強度・変形を自動評価
       ↓
ランキング表示 → 設計者が最終選択
       ↓
CADデータとして出力
```

**使えるツール**

| ツール | 特徴 | 価格帯 |
|---|---|---|
| Autodesk Fusion 360 | ジェネレーティブデザイン機能内蔵 | 月額課金 |
| Ansys Discovery | シミュレーション連携が強力 | エンタープライズ |
| nTopology | 複雑なラティス構造も生成可能 | エンタープライズ |

**期待効果**
- 軽量化：従来比 20〜40% 軽量化の実績あり（航空・自動車業界）
- 設計時間：初期案出しが数週間 → 数時間に短縮

### アイデア：CAD操作の自然言語化

**概要説明**

「ここに直径10mmの穴を3つ、等間隔で開けて」と話しかけるだけでCADが操作できる世界。Claude・GPT等のLLMとCADのAPIを組み合わせることで実現しつつある。

**活用フロー**

```
設計者が自然言語で指示
「フランジ部分の肉厚を2mm増やして、強度解析を再実行して」
       ↓
LLMが命令をCAD操作コマンドに変換
       ↓
CADソフトが自動実行
       ↓
結果をチャット形式でフィードバック
```

**現状の課題**
- 複雑な形状操作はまだ難しい
- CAD固有コマンドとLLMの橋渡し部分の開発が必要
- **MCP（Model Context Protocol）** を活用すると実装しやすい

## 2. 品質管理

### アイデア：AI外観検査

**概要説明**

カメラで撮影した製品画像をAIが解析し、傷・汚れ・形状不良を自動検出する仕組み。熟練検査員の「目」をAIに学習させる。

**たとえ話**
> 人間の検査員が1日8時間、10万枚の画像を見続けるのは不可能。AIは疲れず・ブレず・24時間同じ精度で検査する。

**導入フロー**

```
Step 1: 良品・不良品の画像を大量収集（最低500〜1,000枚）
       ↓
Step 2: 画像にラベル付け（どこが不良か、どの種類の不良か）
       ↓
Step 3: AIモデルをトレーニング
       ↓
Step 4: 実ラインにカメラ設置 → リアルタイム判定
       ↓
Step 5: 不良品を自動で弾き出す
```

**使えるツール**

| ツール | 特徴 |
|---|---|
| Landing AI（LandingLens） | 製造業特化の画像AI、少量データでも動く |
| AWS Rekognition | AWSサービスと連携しやすい |
| Azure Custom Vision | Microsoftエコシステムとの親和性が高い |
| NVIDIA Metropolis | エッジでのリアルタイム処理に強い |

**期待効果**
- 検査精度：人間（95%程度）→ AI（99%以上）
- 人件費削減：検査工程の省人化
- 見逃しゼロ：疲労による検査漏れの排除

## 3. シミュレーション・解析

### アイデア：AIサロゲートモデル（代理モデル）

**概要説明**

本来数時間かかるCAEシミュレーション（有限要素解析・流体解析）を、AIが「近似計算」で数秒に短縮する手法。AIが過去のシミュレーション結果を学習し、新しい条件での結果を予測する。

**たとえ話**
> 毎回フルコースを作るのではなく、「この材料と条件ならこの結果になるだろう」とAIが経験則から予測する。料理人がレシピを暗記するイメージ。

**活用フロー**

```
【学習フェーズ】
設計パラメータ（形状・材料・荷重条件）を変えながら
1,000〜10,000ケースのCAEシミュレーションを実行
       ↓
入力パラメータ → 結果の関係をAIが学習

【活用フェーズ】
新しい設計条件を入力
       ↓
AI（代理モデル）が0.1秒で結果を予測
       ↓
良さそうな設計 → 本格CAEで最終確認
```

**期待効果**
- 計算時間：数時間 → 数秒（1,000〜10,000倍の高速化）
- 設計探索：短時間で数万パターンの最適化が可能に

### アイデア：FMEA（故障モード影響解析）支援

**概要説明**

設計仕様や類似製品の過去不具合データをAIに読み込ませ、「どこで・どんな故障が起きうるか」のリストアップと影響度評価の初稿をAIが生成する。

**FMEAとは**

```
部品・工程のリストアップ
       ↓
故障モード（どう壊れるか）を列挙
       ↓
影響度（S）・発生確率（O）・検出難易度（D）を評価
       ↓
RPN（リスク優先数）= S × O × D を算出
       ↓
RPN が高い項目から対策を打つ
```

**AIができること**

| タスク | AI対応度 |
|---|---|
| 過去の不具合事例からの故障モード抽出 | ◎ 得意 |
| 類似製品事例の参照・提案 | ◎ 得意 |
| 影響度・確率の初期スコアリング | ○ 補助的に可能 |
| 対策案の提案 | ○ 補助的に可能 |
| 最終的な技術判断 | ✕ 人間が必須 |

## 5. ナレッジ管理

### アイデア：ベテランナレッジのAI継承

**概要説明**

ベテラン技術者へのヒアリング内容・過去の設計レビュー議事録・トラブルシューティング記録をAIに学習させ、「社内専門家AI」を作るアイデア。若手が「なぜこの設計にしたか」を即座に聞ける状態を実現する。

**たとえ話**
> 30年のベテランが退職すると消えてしまう「頭の中の知恵」を、退職前にAIに全部コピーするイメージ。

**構築フロー**

```
ベテラン技術者の知恵を文書化
（設計判断の理由・過去トラブル事例・ノウハウ集）
       ↓
RAG（検索拡張生成）システムに組み込む
       ↓
社内ナレッジAIチャットボットとして公開
       ↓
若手エンジニアが自然言語で質問
「なぜこの部品はアルミではなくSUS304を使っているの？」
       ↓
過去の設計判断根拠をAIが即答
```

**RAGとは**
RAG（Retrieval-Augmented Generation）：AIが回答する前に社内ドキュメントを検索し、その内容を根拠にして答える仕組み。「AIのハルシネーション（でたらめ回答）」を大幅に減らせる。

## 6. 予知保全

### アイデア：設備の予知保全AI

**概要説明**

製造設備に取り付けたセンサー（振動・温度・電流・音）のデータをAIが継続的に分析し、「このパターンは2週間後に故障する」と予測する仕組み。計画外停止をなくし、生産ラインを止めない。

**たとえ話**
> 健康診断で「このまま放置すると1年後に心臓病になる可能性80%」と言われるイメージ。問題が起きてから対処するのではなく、なる前に対処する。

**予知保全のレベル**

```
レベル1：事後保全
壊れてから修理する（コスト最大、停止時間最大）
       ↓
レベル2：定期保全
決まった期間ごとに交換・点検する
       ↓
レベル3：状態監視保全
センサーで状態を監視し、閾値を超えたら対処
       ↓
レベル4：予知保全（AI活用）← 目指すゴール
AIが故障を事前に予測して計画的に対処
```

**実装に必要なもの**

| 必要なもの | 内容 |
|---|---|
| センサー | 振動・温度・電流・音響センサー |
| データ収集基盤 | IoTゲートウェイ、クラウド送信 |
| AI分析基盤 | 異常検知モデル（教師あり or 異常検知） |
| アラートシステム | スマホ・メール・ダッシュボードへの通知 |

## 8. 組み込みソフトウェア開発 × AI

> 組み込みソフトウェアは「動いて当たり前、壊れたら許されない」の世界。AIはそのシビアな現場でこそ力を発揮できる。

### アイデア：コーディング規約準拠のコード生成補助

**概要説明**

組み込み開発では MISRA-C / MISRA-C++ などの厳格なコーディング規約が義務付けられることが多い。AIにその規約を事前に覚えさせることで、規約に準拠したコードの初稿生成・違反チェックを自動化できる。

**たとえ話**
> 新人エンジニアに「うちのルールブックを全部読んでから書いて」と渡すイメージ。AIなら一瞬で読み込み、ルールを守ったコードを出力できる。

**CLAUDE.md を使った規約の仕込み方**

```markdown
# CLAUDE.md（プロジェクトルート）

## コーディング規約
- MISRA-C 2012 に準拠すること
- 動的メモリ確保（malloc/free）は禁止
- 関数は1つで1つの責務のみ、50行以内
- 全ての関数にDoxygenコメントを付与
- グローバル変数は最小限に、必ずコメントで用途を明記

## 割り込み処理のルール
- 割り込みハンドラ内でブロッキング処理を行わないこと
- 共有データへのアクセスは必ずクリティカルセクションで保護
- 割り込みハンドラはできる限り短く（フラグを立てるだけ）
```

このファイルを置くだけで、Claude Codeがプロジェクト固有のルールを自動的に読み込み、コード生成・レビュー時に適用する。

**AIでできること / できないこと**

| タスク | AI対応度 | 補足 |
|---|---|---|
| 規約準拠コードの初稿生成 | ◎ | CLAUDE.mdで規約を事前定義 |
| 静的解析（MISRA違反）の一次チェック | ◎ | 人間レビューの前段として有効 |
| Doxygenコメントの自動生成 | ◎ | 関数の意図・引数・戻り値を記述 |
| リアルタイム制約の保証 | ✕ | タイミング解析は人間＋専用ツールが必須 |
| ハードウェア依存の動作保証 | ✕ | 実機テストは省略不可 |

### アイデア：テストコード・テスト仕様書の自動生成

**概要説明**

関数の仕様（入力・出力・境界値）をAIに読み込ませ、ユニットテストコードとテスト仕様書の初稿を自動生成する。テストケースの抜け漏れを減らし、テスト設計の時間を大幅に削減できる。

**活用フロー**

```
設計者が関数仕様を記述
（入力範囲・正常系・異常系・境界値）
       ↓
AIがテストケース一覧を生成
（正常値・最大値・最小値・範囲外・NULLポインタ等）
       ↓
AIがCUnit / Google Test形式のテストコードを生成
       ↓
エンジニアがレビュー・追加・修正
       ↓
CI/CDパイプラインに組み込んで自動実行
```

**生成例（Cユニットテスト）**

```c
/* AIが生成したテストコードのイメージ */
void test_calculate_checksum_normal(void) {
    /* 正常系：有効なデータ配列 */
    uint8_t data[] = {0x01, 0x02, 0x03};
    CU_ASSERT_EQUAL(calculate_checksum(data, 3), 0x06);
}

void test_calculate_checksum_null_pointer(void) {
    /* 異常系：NULLポインタ渡し */
    CU_ASSERT_EQUAL(calculate_checksum(NULL, 3), CHECKSUM_ERR_NULL);
}

void test_calculate_checksum_zero_length(void) {
    /* 境界値：長さ0 */
    uint8_t data[] = {0x01};
    CU_ASSERT_EQUAL(calculate_checksum(data, 0), 0x00);
}
```

### アイデア：ADRで設計判断を記録し、AIに渡す

**概要説明**

「なぜこのタスク優先度にしたか」「なぜMutexではなくセマフォを選んだか」といった設計判断をADR（Architecture Decision Record）として記録する。これにより、後から参加したメンバー（人間もAIも）が「なぜそう作ったか」を把握できる。

**組み込み開発でのADR例**

```markdown
# ADR-003：タスクスケジューラの選定

## 決定事項
FreeRTOS を採用する（μITRON は採用しない）

## 理由
- ターゲットMCU（STM32H7）に対するFreeRTOSの公式サポートが充実
- チーム内にFreeRTOSの経験者が複数いる
- オープンソースでライセンスコストゼロ

## 却下した選択肢
- μITRON：国内実績は多いが、海外向け展開時のサポートが薄い
- Azure RTOS：統合は魅力だが習得コストが高い
```

このADRをCLAUDE.mdから参照させることで、AIが「なぜFreeRTOSか」を理解した上でコード生成・レビューを行える。

## 9. AI抵抗感のあるチームへの導入戦略

> AIの導入で最も難しいのは、技術ではなく「人」の問題。ベテランが持つ不安や懸念は正当なものが多く、それを無視した強引な導入は必ず失敗する。

### まず知っておく：ベテランが抵抗する本当の理由

「AIが怖い・嫌い」ではなく、以下の正当な懸念が背景にあることが多い。

| 懸念 | 裏にある本音 |
|---|---|
| 「AIの出力は信用できない」 | 長年培った品質基準を下げたくない |
| 「責任の所在が曖昧になる」 | 自分のプロがAIに取って代わられたくない |
| 「覚えることが増える」 | 今のやり方で十分うまくいっている |
| 「若手がAI頼りになる」 | 考える力・技術力が失われることへの危惧 |

これらは「変化への抵抗」ではなく「品質・技術への誇り」から来ている。その誇りを活かす方向でAIを位置付けることが突破口になる。

---

### 戦略1：「AIはベテランの仕事を奪わない、雑用を奪う」と示す

**概要説明**

最初のターゲットは、ベテランが「時間の無駄」と感じているが避けられない雑務。AIがその雑務を引き受けることで、ベテランは本来の仕事（設計・判断・レビュー）に集中できる。

**狙うべき「雑務」の例**

| 雑務 | AIへの置き換え方 |
|---|---|
| Doxygenコメント書き | AIがコードを読んでコメントを自動生成 |
| テスト仕様書の初稿作成 | 仕様からAIがテストケースを列挙 |
| 会議の議事録・アクション整理 | Microsoft Copilot for Teams が自動生成 |
| 設計変更の影響調査 | AIがコード全体を検索して影響箇所を列挙 |
| 過去資料の検索 | RAGシステムで社内文書をAIが検索 |

**ポイント**
「AIがコードを書く」ではなく「AIがベテランの準備を手伝う」から始める。脅威ではなく道具として見せることが大事。

---

### 戦略2：ベテランをAIのルール作りに巻き込む

**概要説明**

CLAUDE.mdやコーディングルールの定義を、AIに詳しい若手だけでなくベテランが主導して作る。「自分たちのルールでAIを縛る」という構図にすることで、ベテランが主体的に関われる。

**たとえ話**
> 新入社員を採用するとき、ルール作りに先輩社員が関わるのと同じ。「AIはうちのルールに従わせる」という立場にすれば、脅威ではなく「管理できる存在」になる。

**進め方**

```
Step 1：ベテランに「AIに守ってほしいルール」をヒアリング
         （「AIにやってほしくないこと」も含めて）
       ↓
Step 2：ヒアリング内容をCLAUDE.mdに落とし込む
       ↓
Step 3：ベテランの前でAIがルール通りに動くことを実演
       ↓
Step 4：ルール違反があれば即フィードバックしてCLAUDE.mdを更新
         → ベテランが「AIを育てる」感覚を持てる
```

**CLAUDE.mdの例（ベテラン主導で作成）**

```markdown
## チームルール（○○さんのレビュー基準より）
- 割り込みハンドラ内でOSAPIを呼ばないこと
- タイムアウト処理を必ず実装すること（無限ループ禁止）
- HALドライバを直接叩かず、必ずラッパー関数を経由すること
- ポインタのNULLチェックは必ず実施すること
```

---

### 戦略3：小さな成功体験を積み上げる（スモールスタート）

**概要説明**

最初から大きなシステムや重要な機能にAIを使わない。失敗しても影響が小さい・リカバリが容易なところから始めて、「AIって思ったより使える」という実感をチームに積み上げる。

**スモールスタートのロードマップ**

```
Phase 1（最初の1ヶ月）：個人で使ってみる
 - 自分の担当コードのDoxygenコメントをAIに生成させてみる
 - 会議のアクションアイテムをCopilotに整理させてみる
 - 効果を個人的に計測する（何分節約できたか）
       ↓
Phase 2（2〜3ヶ月目）：チームに共有する
 - 成功体験を朝会やSlackで共有（数字で示す）
 - 希望者だけが使う任意参加の形でスタート
 - 強制しない・比較しない
       ↓
Phase 3（4〜6ヶ月目）：プロセスに組み込む
 - CI/CDへのAIレビュー追加（ルールはベテランが定義）
 - テスト仕様書作成フローにAIを組み込む
 - ベテランが「品質ゲート」として最終確認する役割を担う
       ↓
Phase 4（6ヶ月以降）：標準化
 - CLAUDE.mdをチーム公式ドキュメントとして整備
 - 新メンバーへのAI活用オンボーディング整備
```

---

### 戦略4：「AIが間違えたら人間が直す」の文化を作る

**概要説明**

AIは必ず間違える。その前提を最初から共有し、「AIの出力は必ず人間がレビューする」というルールを明文化する。これにより、ベテランの「AIを信用できない」という懸念に正面から応える。

**「AIの間違いはむしろ価値がある」という視点**

組み込み開発では、「なぜこのコードはこう書くべきか」という暗黙知が多い。AIが間違えたとき、ベテランが「それはこういう理由でダメだ」と指摘する場面が、チームのナレッジ共有の機会になる。

```
AIが間違ったコードを提案
       ↓
ベテランが「これはダメ、なぜなら〜」と指摘
       ↓
その理由をADR・CLAUDE.mdに記録
       ↓
次回からAIは同じ間違いをしない
       ↓
ベテランの知恵がチームの資産になる
```

---

### 戦略5：指標で語る（感情論にしない）

**概要説明**

「AIは良い・悪い」の感情論ではなく、数字で語る。AI導入前後で何がどう変わったかを計測・記録し、判断の材料をデータにする。

**計測すべき指標の例**

| 指標 | 計測方法 |
|---|---|
| ドキュメント作成時間 | AI導入前後で同種作業の所要時間を比較 |
| コードレビュー指摘件数の変化 | AIレビュー導入前後でのレビュー指摘数 |
| MISRA違反の検出数 | 静的解析のAI補助前後で比較 |
| テストカバレッジ | AIによるテスト生成導入前後 |
| 開発者の主観評価 | 月次アンケート（5段階評価） |

**重要な原則**
- 個人の生産性を比較・ランキング化しない（心理的安全性の破壊につながる）
- チーム全体の改善を示す指標に絞る
- 悪い数字が出ても正直に共有し、改善策を一緒に考える

---

### まとめ：チーム導入の心得

```
✕ やってはいけないこと
─────────────────────────────
「AIを使わない人は時代遅れ」という雰囲気を作る
ベテランの承認なく重要なコードにAIを使う
AI出力をレビューなしで本番に入れる
導入効果を個人の成績評価に結びつける

✅ やるべきこと
─────────────────────────────
ベテランをルール作りの主役にする
雑務の削減から始めて「楽になった」を体感させる
間違いを責めず、学習の機会として記録する
数字で効果を示し、感情論にしない
「AIは道具、判断は人間」の原則を守り続ける
```
