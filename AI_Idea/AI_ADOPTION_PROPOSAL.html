課題別：背景と推進ステップ

課題1：コードレビュー・効率化

背景

有識者へのレビュー集中とレビュー待ちの長期化が開発速度のボトルネックになっている。また、レビュー基準が有識者の頭の中にあり、担当者によって指摘内容にばらつきが生じている。

解決後の姿

AIがコードを解析してフィードバックを自動生成する。設計ルール・チェック基準をAIへのInstructions（CLAUDE.mdなど）として文書化することで、属人化を解消する。有識者はAIが対応できない高難度の判断のみを担当する。

推進ステップ

Step 1【1ヶ月目】
  有識者に「よく指摘するパターン」をヒアリング
  → レビューチェックリストとして文書化

Step 2【2ヶ月目】
  チェックリストをCLAUDE.md（AIへの指示書）に落とし込む
  → 1つのリポジトリでAI自動レビューを試験導入
  → PRを出すとAIがコメントを自動投稿する状態を作る

Step 3【3ヶ月目】
  AI指摘と有識者指摘を比較して精度を評価
  → 漏れているルールをCLAUDE.mdに追記

Step 4【4ヶ月目以降】
  全リポジトリへ展開
  → 有識者のレビュー工数を定量計測して改善を継続

計測指標

指標 目標
レビュー待ち時間 現状比 50% 削減
有識者のレビュー工数 現状比 40% 削減
規約違反の後工程発覚件数 現状比 70% 削減

課題2：評価制度の確立

背景

有識者が行ってきた評価の基準・内容が明文化されておらず、見える化されていない。評価者によって品質判断にばらつきがあり、チーム全体での品質水準の統一が難しい状態にある。

解決後の姿

過去コードの評価基準を定量的なルールとして確立する。新規コードに対してそのルールを一貫して適用することで、評価の属人性をなくす。評価結果をアナリティクスとして可視化し、チーム全体の品質傾向を把握できるようにする。

推進ステップ

Step 1【1ヶ月目】
  有識者に「品質が高いコード」「低いコード」の判断基準をヒアリング
  → 定量的な評価項目に変換（複雑度・テストカバレッジ・規約遵守率など）

Step 2【2ヶ月目】
  過去コードに対して評価ルールを適用し、スコアリングを試験実施
  → 有識者の直感評価と乖離がないか確認・調整

Step 3【3ヶ月目】
  新規コードへの評価ルール適用をCI/CDに組み込む
  → スコアが基準を下回るとPRがブロックされる状態を作る

Step 4【4ヶ月目以降】
  評価結果のダッシュボード化（コードメトリクスの可視化）
  → 月次でチームの品質トレンドを共有

課題3：テストの自動化

背景

コーディング規約チェック・ビルド・テスト実行が手動で行われており、実行漏れや環境差異による問題が発生している。テスト工数が大きく、エンジニアの時間がコーディング以外の作業に取られている。

解決後の姿

GitHub Actionsを使ったCI/CDパイプラインを整備し、コードがpushされるたびにビルド・規約チェック・テストが自動実行される。人が手動で実行する作業をなくし、テストの抜け漏れを防ぐ。

パイプラインの構成

開発者が git push
       ↓
GitHub Actions 起動
       ↓
① クロスコンパイル（ターゲットMCU向けビルド）
② コーディング規約チェック（MISRA-C等）
③ ユニットテスト自動実行
④ カバレッジ計測・レポート生成
⑤ AIによるコードレビューコメントをPRに自動投稿
       ↓
全項目通過 → レビュワーへ通知

推進ステップ

Step 1【1ヶ月目】
  現在の手動テスト工程を洗い出し、自動化の優先順位を決める

Step 2【2ヶ月目】
  最小構成のパイプラインを構築（ビルドの自動化のみ）
  → まず「動く」状態を最速で作る

Step 3【3ヶ月目】
  規約チェック・ユニットテストをパイプラインに追加

Step 4【4ヶ月目以降】
  AIレビューをパイプラインに統合。全リポジトリへ展開

課題4：開発環境の改善

背景

コーディング作業そのものに多くの時間がかかっている。コードを変更した際の影響範囲調査（どのファイル・モジュールに波及するか）が手作業で、変更のたびに大きな工数を要している。

解決後の姿

AIコード補完（GitHub Copilot等）をIDEに統合し、コーディング時間を削減する。影響範囲調査もAIが対象ファイルを自動列挙することで、調査工数を大幅に削減する。

推進ステップ

Step 1【1ヶ月目】
  VS Code + GitHub Copilot を全員がインストールして2週間試用
  → 試用後に「どの作業で時間が短縮できたか」を共有

Step 2【2ヶ月目】
  CLAUDE.md（プロジェクトルール）を整備
  → AIがプロジェクト固有のコーディング規約を理解した状態にする

Step 3【3ヶ月目】
  影響調査へのAI活用を試験導入
  → 変更ファイルを入力するとAIが影響箇所を列挙するフローを作る

Step 4【4ヶ月目以降】
  ツールの標準化・ライセンスの整備。効果を定量計測して継続改善

課題5：OSS変更の効率化

背景

使用しているOSSのバージョンアップ対応が手作業で行われており、差分の確認・影響範囲の調査・コード修正のすべてに大きな工数がかかっている。OSSアップデートを先送りにすることでセキュリティリスクが蓄積するという問題もある。

解決後の姿

OSS更新時の差分解析・影響調査・修正箇所の特定をAIが支援する。アップデート対応のリードタイムを短縮し、セキュリティパッチへの対応速度を上げる。

推進ステップ

Step 1【1ヶ月目】
  現在使用しているOSSのリストを整備
  → 更新頻度・影響範囲の大きさで優先順位をつける

Step 2【2ヶ月目】
  OSSのリリースノート・差分をAIに読み込ませて影響箇所を列挙するフローを試作
  → 1つのOSSで試験的に実施

Step 3【3ヶ月目】
  影響調査結果をもとにAIが修正候補コードを生成するフローを追加
  → 人間がレビュー・修正して適用

Step 4【4ヶ月目以降】
  対象OSSを拡大。定期的なアップデートチェックを自動化

課題6：設計情報の構造化

背景

設計情報がWordドキュメント・Excelシート・ソースコードのコメント・Wikiなどに分散しており、必要な情報にたどり着くまでに時間がかかる。設計意図がコードに反映されているかどうかの確認も困難な状態にある。

解決後の姿

散在する設計情報をAIが収集・構造化してナレッジベースを構築する。エンジニアは自然言語で「この機能の設計意図は？」と問いかけるだけで必要な情報を即座に得られる。

推進ステップ

Step 1【1ヶ月目】
  設計情報の在処を棚卸し（どこに何があるかのマッピング）
  → 最も参照頻度が高いドキュメントを特定

Step 2【2ヶ月目】
  RAGシステム（ドキュメント検索AI）のPoC構築
  → 主要ドキュメントを読み込ませてQ&Aを試験

Step 3【3ヶ月目】
  特定チームでの試験運用
  → 「設計情報を探す時間」を計測して効果を確認

Step 4【4ヶ月目以降】
  対象ドキュメントを拡大。コードとドキュメントのリンク化も検討

課題7：JIRA活用の効率化

背景

JIRAのチケット管理が担当者ごとに異なるやり方で運用されており、プロジェクト全体の進捗状況・リスクの把握が困難な状態にある。チケットの記載内容もばらつきがあり、情報の質が安定しない。

解決後の姿

AIとJIRAを連携させ、チケットの自動分類・進捗サマリーの自動生成・リスクのある課題の自動検知を実現する。マネージャーが毎朝JIRAを手動で確認しなくても、AIが状況をサマリーして通知する状態にする。

推進ステップ

Step 1【1ヶ月目】
  JIRAの現在の運用状況を整理
  → チケットのフィールド定義・ステータスの統一ルールを策定

Step 2【2ヶ月目】
  AIによる進捗サマリーを試験導入
  → JIRAのデータを週次でAIがサマリーしてSlack/Teamsに投稿

Step 3【3ヶ月目】
  チケットの自動分類・優先度スコアリングをAIで試験実施

Step 4【4ヶ月目以降】
  リスク検知の自動化（期限超過の予兆・ボトルネックの早期発見）

課題8：設計ドキュメントの整備

背景

設計書の作成・更新がコーディング・テストに優先度を奪われて後回しになりがち。SharePoint上に最新版と旧版が混在しており、どれを参照すればよいか判断が難しい状態にある。

解決後の姿

AIがコードや変更差分から設計書のドラフトを自動生成することで、ドキュメント作成の工数を削減する。SharePointとCI/CDを連携させ、コードマージのタイミングでドキュメントが自動更新される状態を実現する。

推進ステップ

Step 1【1ヶ月目】
  SharePoint上のドキュメントを棚卸し
  → 最新版の特定・旧版のアーカイブ化・命名規則の統一

Step 2【2ヶ月目】
  AIによるDoxygenコメント自動生成を全エンジニアに試験導入
  → コードからAPIドキュメントが自動生成される状態を作る

Step 3【3ヶ月目】
  設計変更時の仕様書ドラフト生成をAIで補助
  → 変更差分を入力するとAIが更新箇所のドラフトを出力

Step 4【4ヶ月目以降】
  CI/CDとSharePointを連携させたドキュメント自動更新フローの構築

ドキュメント構造の整理案

SharePoint/
├── 現行仕様書/       ← CI/CDが自動更新
├── 設計判断記録(ADR)/ ← 変更理由の蓄積
├── アーカイブ/       ← 過去バージョン
└── ナレッジベース/   ← AIの学習元ドキュメント

課題9：AI Agentの生成（後発組）

背景

チームの一部メンバーはまだAIをチャットツールとして使う段階にとどまっており、業務フローへのAI組み込み（AIエージェント化）が実現できていない。AIエージェントの作り方・使い方のスキルが不足している。

解決後の姿

AIエージェントを自分で作れるスキルを習得し、自分の業務の一部をAIエージェントに任せられる状態を実現する。繰り返し作業・定型作業の自動化によって本来の開発業務に集中できるようになる。

AIエージェントとは

通常のAI（チャット）：
人間が都度質問 → AIが回答 → 人間が次の行動を判断

AIエージェント：
人間が目標を与える → AIが自律的にツールを使いながら作業を進める
→ 人間は結果だけを確認・承認する

推進ステップ

Step 1【1ヶ月目】
  後発組メンバーのAIリテラシーの現状把握
  → まず日常業務でのAI活用（コード補完・ドキュメント生成）を定着させる

Step 2【2ヶ月目】
  AIエージェントのハンズオン研修を実施（月1〜2回）
  → Claude Code / Copilot Studio でシンプルなエージェントを作る体験

Step 3【3ヶ月目】
  各自が担当業務の中で「自動化したい繰り返し作業」を1つ選んでAIエージェント化
  → 小さな成功体験を積む

Step 4【4ヶ月目以降】
  作成したAIエージェントをチームで共有・横展開
  → エージェントのカタログを整備して再利用を促進

課題10：コード生成

背景

コーディング作業そのものに多くの時間を要しており、仕様書や設計ドキュメントを参照しながら手動でコードを書く作業が大部分を占めている。特に定型的な処理（初期化・エラーハンドリング・ログ出力など）の実装に時間がかかっている。

解決後の姿

AIが仕様・設計ドキュメントを参照しながらコードの初稿を自動生成する。エンジニアはAIが生成したコードをレビュー・修正する役割に移行し、コーディングの生産性が向上する。

推進ステップ

Step 1【1ヶ月目】
  AIコード生成ツール（GitHub Copilot / Claude Code）を全員が試用
  → 「AIが得意な作業」「人間が判断すべき作業」の境界を把握する

Step 2【2ヶ月目】
  CLAUDE.md にプロジェクトのコーディング規約・アーキテクチャを定義
  → AIが正しいコードを生成できる前提条件を整える

Step 3【3ヶ月目】
  定型処理（初期化・エラーハンドリング・ドライバ呼び出しなど）の
  コード生成テンプレートをAIと協力して整備

Step 4【4ヶ月目以降】
  設計ドキュメントを入力するとAIがコードの初稿を生成するフローを確立
  → 生成コードの品質をCI/CDで自動検証する仕組みと連動させる

AIを導入する際の基本的な考え方

AIは有識者を置き換えるものではない

有識者が長年かけて習得してきた判断力・経験・ノウハウは、AIが簡単に代替できるものではない。AIが担うのは「調べる・書く・チェックする」といった作業の自動化であり、「何が正しいか判断する」役割は引き続き有識者が担う。

小さく始めて、確実に積み上げる

全課題を一斉に進めるのではなく、影響範囲が小さく効果が出やすいところから始めて、成功体験を積み上げながら横展開する。失敗しても被害が小さい場所でPoC（試験導入）を行い、そこで得た知見を次に活かす。

有識者がルールを作る

AIへの指示書（CLAUDE.mdなど）やレビューチェックリストは、AI担当者だけでなく有識者が主導して作成する。「AIを管理する側」として関わることで、AI導入への主体的な参加を促す。
