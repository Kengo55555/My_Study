## 1. Cloudflare

**概要説明**

**インターネットを速く・安全にする世界最大級のクラウドネットワーク**。Webサイトとユーザーの間に入って、攻撃を防ぎ、表示を速くする。

**主な機能（3本柱）**

| 機能 | 役割 |
|---|---|
| **CDN** | 世界中のサーバーから配信して高速化（CDN市場シェア約80%） |
| **WAF** | Webアプリケーション・ファイアウォール |
| **DDoS対策** | 大規模攻撃から保護（最大197Tbps） |

**その他の代表機能**

| 機能 | 概要 |
|---|---|
| 1.1.1.1 | 無料・高速DNSサービス（プライバシー重視） |
| Workers | エッジで動くサーバーレス環境 |
| R2 | S3互換のオブジェクトストレージ |
| D1 | SQLiteベースのエッジDB |
| Zero Trust | SASE/ゼロトラストセキュリティ |
| メールルーティング・ドメインレジストラ | ドメイン・メール管理 |

**Claudeとの連携**

公式Cloudflare MCPコネクタで、自然言語でWorkers・D1・R2・KVを操作可能。

**類似サービス**

| サービス | 特徴 |
|---|---|
| AWS CloudFront / WAF / Shield | AWS純正、細かい設定が可能 |
| Akamai | エンタープライズ向けの老舗CDN |
| Fastly | 高速エッジコンピューティング |

## 2. CI/CD

**概要説明**

ソフトウェアの**ビルド・テスト・デプロイの流れを自動化**する手法。

| 略 | 意味 | 何をする |
|---|---|---|
| **CI** | Continuous Integration（継続的インテグレーション） | コードを統合するたびに自動でビルド・テスト |
| **CD** | Continuous Delivery / Deployment | リリース直前まで（デリバリー）／本番反映まで（デプロイメント）自動化 |

**典型的な流れ**

```
開発者がGitにpush
    ↓
CIが自動でビルド & テスト
    ↓
テスト合格でマージ可能
    ↓
CDがステージング → 本番へ自動デプロイ
```

**主なツール**

| ツール | 特徴 |
|---|---|
| **GitHub Actions** | 最もシェアが大きい |
| GitLab CI/CD | GitLabと統合 |
| CircleCI | クラウドCI/CDサービス |
| Jenkins | 老舗、自由度が高い |
| Argo CD | Kubernetes向け |

**AI駆動開発とCI/CDの関係**

1. **AIは速く大量にコードを書く** → CI/CDがないと検証が追いつかない
2. **AIは間違える** → CI/CDが「品質ゲート」として機能
3. **AIエージェント自身がCI/CDを使う・触る時代** → GitHub Actionsが最頻度

> AI駆動開発が「コードを書く速度」を上げ、CI/CDが「安全に届ける速度」を上げる。両方が揃って初めて、AI開発の生産性がユーザー価値に変わる。

## 3. Claude Codeのインストール確認

**ターミナルで確認**

```bash
claude --version
```
- バージョン番号が表示 → インストール済み ✅
- `command not found` → 未インストール ❌

**インストール方法**

```bash
npm install -g @anthropic-ai/claude-code
```
※ Node.js（v18以上）が必要。`node --version`で確認可能。

**Claude Desktop（チャット型）との違い**

| ツール | 説明 |
|---|---|
| Claude Code | ターミナル/CLIで動く、コーディング特化のエージェント |
| Claude Desktop | チャットUI、一般的な用途 |

## 4. Docker Daemonに接続できないエラー

**症状**

```
failed to connect to the docker API at unix:///var/run/docker.sock
```
`"Server": null`が出ている → Dockerデーモン（エンジン）が起動していない。

**対処法**

```bash
# 1. Docker Desktop を起動（メニューバーの🐳が緑になるまで待つ）

# 2. 動作確認
docker version

# 3. ソケットパスがない場合（macOS既知の問題）
sudo ln -sf "$HOME/.docker/run/docker.sock" /var/run/docker.sock

# 4. Docker Desktop の Settings → Advanced で
#    「Allow the default Docker socket to be used」を ON
```

**VS Code側で再試行**

- `Cmd+Shift+P` → `Dev Containers: Reopen in Container`

## 5. macOS Gatekeeper：permission denied 対応

**症状**

```bash
$ ./teams-transcription-dl
zsh: permission denied: ./teams-transcription-dl
```
macOSのGatekeeper（セキュリティ機能）が**開発元不明のアプリ**をブロックしている状態。

**解除方法**

```bash
# 検疫属性を解除
xattr -d com.apple.quarantine /path/to/file

# 実行権限を付与
chmod +x /path/to/file

# 実行
./file
```

**それでもダメなら**

1. システム設定 → プライバシーとセキュリティ
2. 「○○はブロックされました」 → 「このまま許可」
3. 再度実行

> ⚠️ **注意**：出所不明のバイナリは実行前に必ず信頼性を確認すること。

## 6. GitHub ActionsでCI/CDパイプラインを構築する

**概要説明**

**GitHub Actions** は、GitHubリポジトリ内に設定ファイル（YAML）を1枚置くだけで、ビルド・テスト・デプロイを自動化できるCI/CDサービス。「料理のレシピ」のように、一連の工程を毎回手作業ではなくボタン一つ（＝コードのpush）で再現できる。

| 用語 | 意味 |
|---|---|
| Workflow（ワークフロー） | 自動処理のまとまり全体 |
| Event（イベント） | ワークフローを起動するきっかけ（pushやPR作成など） |
| Job（ジョブ） | ワークフロー内の処理のまとまり |
| Step（ステップ） | ジョブ内の個別の処理 |
| Action（アクション） | 再利用可能な処理の部品（公式・サードパーティ製） |
| Runner（ランナー） | 処理を実際に実行する仮想マシン |

**使い方・使用例**

#### ステップ1：ワークフローファイルを置く場所を作る

リポジトリのルートに次のフォルダ構成を作り、`.github/workflows/` の中にYAMLファイルを置く（これがルール）。

```
あなたのプロジェクト/
├── .github/
│   └── workflows/
│       └── ci-cd.yml   ← 設定ファイル
├── src/
└── package.json
```

#### ステップ2：CI（自動テスト）の最小構成

Node.jsプロジェクトを例に、最もシンプルなCIワークフロー。

```yaml
name: CI Pipeline

on:
  push:
    branches: [ main ]      # mainブランチへのpush時
  pull_request:
    branches: [ main ]      # mainブランチへのPR時

jobs:
  test:
    runs-on: ubuntu-latest  # GitHubが用意する仮想マシン
    steps:
      - name: コードをチェックアウト
        uses: actions/checkout@v4

      - name: Node.jsをセットアップ
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: 依存パッケージをインストール
        run: npm ci

      - name: Lintチェック
        run: npm run lint

      - name: テスト実行
        run: npm test

      - name: ビルド
        run: npm run build
```

| 部分 | 意味 |
|---|---|
| `on:` | いつ実行するかを指定（pushやPR） |
| `runs-on:` | 実行OSの指定（Ubuntu最新版） |
| `uses:` | 既製のActionを呼び出す |
| `run:` | ターミナルで打つコマンドをそのまま書ける |

#### ステップ3：CD（自動デプロイ）を追加

テスト合格後、Vercelに自動デプロイする例。

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm test
      - run: npm run build

  deploy:
    needs: test                              # testジョブ成功後に実行
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'      # mainブランチの時だけデプロイ
    steps:
      - uses: actions/checkout@v4
      - name: Vercelにデプロイ
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          vercel-args: '--prod'
```

| ポイント | 内容 |
|---|---|
| `needs:` | ジョブの依存関係。失敗したらデプロイされない |
| `if:` | 条件式。指定ブランチだけ動かしたい時に使う |
| `${{ secrets.X }}` | APIキーなどはコードに直接書かず、Secrets機能で管理 |

#### ステップ4：Secrets（機密情報）の登録

1. GitHubリポジトリの「Settings」タブ
2. 左メニュー「Secrets and variables」→「Actions」
3. 「New repository secret」ボタンで名前と値を登録（例：`VERCEL_TOKEN`）

YAML側の `${{ secrets.VERCEL_TOKEN }}` と**完全一致する名前**で登録する必要がある。

#### ステップ5：動作確認

```bash
git add .github/workflows/ci-cd.yml
git commit -m "CI/CDパイプラインを追加"
git push origin main
```

リポジトリの「Actions」タブで実行状況がリアルタイムに表示される。緑のチェック＝成功、赤のバツ＝失敗（クリックでログ確認）。

**初心者がつまずきやすいポイント**

| 落とし穴 | 対処 |
|---|---|
| インデント崩れ | YAMLは半角スペース2つで揃える（タブはNG） |
| ブランチ名の違い | `main` か `master` か、自分のリポジトリで確認 |
| Secretsの名前不一致 | YAMLとGitHub側で完全一致が必要 |
| 最初から複雑にしすぎる | まずは「pushされたらテストが走る」だけのシンプル構成から |

**類似技術・関連用語**

| ツール | 特徴・選び方 |
|---|---|
| **GitHub Actions** | GitHubに統合、設定ファイルだけで動く。最もシェアが大きい |
| GitLab CI/CD | GitLab使用時の標準 |
| CircleCI | クラウドCI/CDサービス、並列実行が強み |
| Jenkins | 老舗、自由度が高いが構築・運用コストが高め |
| AWS CodePipeline | AWS純正、AWSサービスとの統合が深い |

> 公式の [starter-workflows](https://github.com/actions/starter-workflows) には言語ごとのテンプレが揃っており、コピーして動かすのが最短ルート。
