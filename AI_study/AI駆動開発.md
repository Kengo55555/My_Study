## 1. ハーネスエンジニアリング

**概要説明**

**AIエージェントが安定して正しい仕事をできるよう、モデルの「外側」の環境全体を設計するエンジニアリング手法**。

「**馬具（harness）**」が語源で、馬を制御して意図した方向へ導く比喩。

**進化の流れ**

```
プロンプトエンジニアリング（個別の指示文の工夫）
    ↓
コンテキストエンジニアリング（AIに何を見せるか）
    ↓
ハーネスエンジニアリング（環境全体の設計）
```

**4つの構成要素（OpenAIの整理）**

| 要素 | 何をする | 具体例 |
|---|---|---|
| **制約（Constrain）** | できることを制限 | レイヤー境界、依存関係ルール |
| **通知（Inform）** | 何をすべきか伝える | AGENTS.md、ドキュメント |
| **検証（Verify）** | 正しく動いたか確認 | テスト、リンタ、CI |
| **修正（Correct）** | 間違いを直させる | フィードバックループ |

**実例**

| 実例 | 概要 |
|---|---|
| AGENTS.md / CLAUDE.md | AIへの指示書を「目次」として活用（百科事典化を避ける） |
| 3回ルール | 同じ違反が3回発生したらルールを1段階強化（お願い → リンタ警告 → CI強制） |
| エージェント同士のコードレビュー | 人間のレビューがボトルネックにならないように |
| リポジトリをSSOTに | Slack・口伝はAIから見えないため、リポジトリを唯一の真実のソースにする |

**有名な事例**

OpenAIのCodexチームが**3人・5ヶ月で約100万行のコード**を出荷（人間が手書きしたコードはゼロ）

## 2. ADR（Architecture Decision Record）

**概要説明**

「なぜその設計判断をしたか」を記録した小さなマークダウンファイルの集合。AI駆動開発との相性が非常によく、AIに「過去の意思決定の文脈」を渡せる。

**典型的なフォルダ構成**

```
docs/
├── REQUIREMENTS.md   # 要件（何を作るか）
├── SPEC.md           # 仕様（どう作るか）
└── adr/              # ADR（なぜそう作るか）
    ├── 001-送信方式にE案を採用.md
    └── 002-次ワークセンターの付与対象.md
```

**ADRが役に立つ理由**

| 理由 | 内容 |
|---|---|
| 永続性 | イシューはクローズすると見えなくなることがあるが、ADRはリポジトリ内のドキュメントなのでAIが常に参照できる |
| 文脈の共有 | 後から参加した人（AI含む）でも「なぜこの設計か」がわかる |
| AI親和性 | コンテキストとして渡しやすいMarkdown形式 |

**AI支援開発での運用ヒント**

| ヒント | 内容 |
|---|---|
| イシューには頼らない | 流動的なため最新仕様の出所として頼らない |
| 最新状態はコードで管理 | ソースコードを唯一の真実のソース（SSOT）にする |
| 仕様書の更新タイミング | 大きなマイルストーンでまとめて整備し、毎Issue単位では更新しない |

## 3. Claude Codeでチーム開発を効率化するアイデア

| アイデア | 概要 |
|---|---|
| 設計仕様書の自動生成・同期 | コードからMarkdown/Wordの仕様書を自動生成 |
| GitHub PR レビューの自動化 | GitHub Actionsと組み合わせて自動レビュー |
| 要件 → 設計ドキュメントへの自動変換 | 要件書から詳細設計書の初稿を生成 |
| テスト仕様書・テストコードの自動生成 | 設計書からテスト一式を生成 |
| 管理責任者向けレビューダッシュボード | PR/Issue/Wikiを集約してSlack通知 |
| ナレッジベースの自動蓄積 | バグ修正履歴 → よくあるパターン集に変換 |
| CLAUDE.md でチームルールを標準化 | コーディング規約・ドキュメントルールを一元化 |

**CLAUDE.md の例**

```markdown
# CLAUDE.md

## コーディング規約
- MISRA-C 2012準拠
- 関数は50行以内
- 全関数にDoxygenコメント必須

## ドキュメントルール
- 設計変更時は必ずdocs/フォルダを更新
- 管理責任者承認が必要なファイル：docs/spec/

## レビュー観点
- メモリ確保は必ずNULLチェック
- 割り込み処理内でのブロッキング処理禁止
```

## 4. おすすめの学習順序

AI初心者から段階的にステップアップする順序：

### Step 1：基礎を固める
- Git/GitHub、localhost、ホスティングの概念
- VercelとSupabaseで簡単なアプリをデプロイしてみる

### Step 2：AIを使う
- GitHub CopilotまたはClaude Codeを実際に試す
- 「補完」→「Chat」→「複数ファイル編集」と段階的にレベルアップ

### Step 3：AIをつなぐ
- MCPで自分の作業環境（Notion/Slack/Gmail）と連携
- Notion MCPで議事録自動保存などを体験

### Step 4：AIを高速化する
- CI/CDを導入して自動テスト・自動デプロイを回す
- GitHub Actionsから始めるのが最もシェアが大きく情報も豊富

### Step 5：AIを統制する
- CLAUDE.md・AGENTS.mdでルール化
- ADRで設計判断を記録
- ハーネスエンジニアリングの考え方で環境を整備

### Step 6：AIをホストする
- Ollamaでローカル実行・プライバシー対応の検討
- エッジで動かしたいならCloudflare Workersも視野に

## 5. GitHub CopilotとClaude Codeの使い分け

**概要説明**

両者は「似ているようで別カテゴリ」のAIコーディングツール。例えるなら、**Copilot＝クルーズコントロール**（運転は自分、横で助言）、**Claude Code＝自動運転車**（行き先を伝えると自分で運転）。多くのプロ開発者は**両方を場面ごとに併用**している。

| ツール | 立ち位置 | 主な活躍場面 |
|---|---|---|
| **GitHub Copilot** | IDEに常駐する**助手** | タイピング中の補完、1ファイル内の作業 |
| **Claude Code** | ターミナルで動く**部下（エージェント）** | 複数ファイル横断、リファクタリング、バグ調査 |

**使い方・使用例**

#### 得意分野の比較

| 観点 | GitHub Copilot | Claude Code |
|---|---|---|
| 得意な作業 | インライン補完、1ファイル内の関数、テストのボイラープレート | 複数ファイルのリファクタリング、コードベース全体の理解、ゼロからの構築、バグの追跡 |
| 動作場所 | エディター内（VS Code・JetBrains等） | ターミナル/CLI |
| 操作感 | タブキーで提案を受け入れる | 自然言語でタスクを依頼 |
| コンテキストウィンドウ | 32K〜128K | 最大1Mトークン（中規模リポ全体を一度に把握） |
| 価格帯 | Copilot Pro：月$10 | Claude Code Max：月$100〜 |

#### 機能が重複したときの判断基準

両ツールとも進化してチャット・マルチファイル編集・PRレビューなどが重なってきた。迷った時は**作業のスコープ**で選ぶ。

| 重複機能 | おすすめ判断軸 |
|---|---|
| **コードチャット** | 開いているファイル内の質問は **Copilot**。複数ファイル横断の調査は **Claude Code** |
| **マルチファイル編集** | 3〜5ファイルの軽い修正は **Copilot Agent mode**。10ファイル超や全体理解が必要な変更は **Claude Code** |
| **PRレビュー** | GitHubワークフロー統合がシームレスな **Copilot** |

#### 場面別の選び方

```
1ファイル内で完結？ → Copilot
タイピング中の補完が欲しい？ → Copilot
複数ファイルにまたがる作業？ → Claude Code
バグの原因が分からず広く調査したい？ → Claude Code
プロジェクトをゼロから構築したい？ → Claude Code
タスクを丸ごと任せて結果だけ確認したい？ → Claude Code
```

**初心者へのおすすめ**

| ステップ | 内容 |
|---|---|
| まず始める | **GitHub Copilot**：拡張機能1つで導入、月$10、学生は無料、操作はタブキーだけ |
| 慣れてきたら | **Claude Code**を追加して複数ファイル作業に挑戦 |
| 予算が許せば | 両方併用：Copilotで日常の補完、Claude Codeで複雑タスクという2層構成がプロの標準形 |

**類似技術・関連用語**

| サービス | 特徴 |
|---|---|
| **Cursor** | VS Codeをフォークした専用エディタ。Copilot寄りの体験＋エージェント機能 |
| **Cody（Sourcegraph）** | エンタープライズ向け、社内コードベース検索が強い |
| **Tabnine** | 古参、オンプレ運用も可能 |
| **Aider** | OSSのターミナルエージェント、Claude Code的な使い方 |

> ポイント：Copilotは「**コードを書く速度**」を、Claude Codeは「**コード作業全体**」を引き上げる補完関係。競合ではなく役割分担で考える。
