## 1. 開発の基礎

### 1.1 localhost:3000

**ざっくり何か**
自分のPC上で動いているWebサーバーにアクセスするためのURL。`localhost`は「自分自身のPC」を指す特別なアドレス、`3000`は「窓口の番号（ポート番号）」。

| 部分 | 意味 |
|------|------|
| `http://` | 通信方式 |
| `localhost` | 自分のPC（`127.0.0.1`と同じ） |
| `3000` | ポート番号 |

**使用例**
- React/Next.js/Vueでフロントエンド開発する時：`npm run dev`で自動起動
- Node.js/Expressでバックエンド開発する時
- Dockerでアプリを起動した時の動作確認

**ポイント**
- 自分のPCの中だけで動いているので**他の人からは見えない**
- ポート番号は`3000`の他に`8080`、`5173`（Vite）、`4000`なども一般的

### 1.2 Git/GitHub関連の基本コマンド

**ざっくり何か**
ソースコードのバージョン管理を行う仕組み。`Git`がツール本体、`GitHub`がクラウド上の保管場所。

**初回アップロード時の典型コマンド**

| コマンド | 役割 |
|---|---|
| `git init` | フォルダをGit管理下に置く |
| `git add ファイル名` | コミット対象に追加（ステージング） |
| `git commit -m "メッセージ"` | 変更を記録 |
| `git branch -M main` | ブランチ名を`main`に変更 |
| `git remote add origin URL` | GitHubリポジトリと紐づけ |
| `git push -u origin main` | GitHubへアップロード |

**類似サービス**
- **GitLab** — GitHubの競合、CI/CD機能が強力
- **Bitbucket** — Atlassian社のサービス、Jiraと連携が強い
- **Azure DevOps** — Microsoft、エンタープライズ向け

### 1.3 GitHub EnterpriseとGitHub Team

**ざっくり何か**
GitHubの**有料プラン**で、規模・必要な機能によって階層が分かれる。

| プラン | 対象 | 料金（月額） | 主な機能 |
|---|---|---|---|
| **Free** | 個人・小規模 | $0 | 基本機能 |
| **Team** | 小〜中規模チーム | $4/ユーザー〜 | コードオーナー、保護ブランチ、複数レビュワー、Codespaces |
| **Enterprise** | 大規模組織 | 個別見積もり | SAML SSO、監査ログ、Enterprise Managed Users、データレジデンシー、セルフホスト |

**使い分けのポイント**
- 「**SAML SSO・監査ログ・セルフホスト・データレジデンシー**」が必要かどうかが、TeamとEnterpriseの境目
- Enterpriseは2形態：**Enterprise Cloud**（GitHub側でホスト）と **Enterprise Server**（自社インフラでホスト）

### 1.4 ホスティングとは

**ざっくり何か**
作ったWebサイトやアプリのファイルを、インターネット上のサーバーに置いて、世界中の人がアクセスできるようにするサービス。

**たとえ話**
> 🏠 家（ウェブサイト）を建てても、土地（サーバー）がないと人は訪問できない。ホスティングは「土地を貸してくれる」サービス。

**ホスティングの種類**

| 種類 | 特徴 | 例 |
|---|---|---|
| 共用ホスティング | 1台のサーバーを複数ユーザーで共有 | レンタルサーバー（さくら・XServer） |
| VPS | 仮想専用サーバー | ConoHa、Linode |
| クラウド | 自由に拡張可能な大規模インフラ | AWS、GCP、Azure |
| 静的ホスティング | フロントエンド特化 | **Vercel**、Netlify、Cloudflare Pages |
| BaaS | バックエンド機能込みでホスト | **Firebase**、**Supabase** |

### 1.5 Vercel

**ざっくり何か**
主にフロントエンド（React・Next.jsなど）向けの**クラウドホスティングプラットフォーム**。Next.jsの開発元でもある。

**主な特徴**

| 特徴 | 内容 |
|---|---|
| 簡単デプロイ | GitHubと連携して`push`するだけで自動更新 |
| 高速配信（CDN） | 世界中のサーバーから配信 |
| プレビュー機能 | プルリクエストごとに自動でプレビューURL生成 |
| サーバーレス関数 | バックエンドのAPIもVercel上で実行可能 |
| 無料プラン | 個人利用なら無料 |

**向いている用途**
ポートフォリオ、ブログ、Reactアプリ、Next.jsアプリ全般。

**類似サービス**
- **Netlify** — 静的サイトに強い、似た機能
- **Cloudflare Pages** — Cloudflareエッジ網で高速
- **AWS Amplify** — AWS純正、自由度高め
- **Firebase Hosting** — Googleのサービスと連携しやすい

### 1.6 Supabase

**ざっくり何か**
**「Firebaseのオープンソース代替」** として作られたBackend-as-a-Service（BaaS）。アプリのバックエンドに必要な機能をまるごと提供する。

**Firebaseとの比較**

| 項目 | Supabase | Firebase |
|---|---|---|
| 提供元 | オープンソース | Google |
| データベース | **PostgreSQL（SQL）** | Firestore（NoSQL） |
| 料金 | 無料枠が広め | 無料枠あり |
| オープンソース | ✅ 自前サーバーも可 | ❌ |

**主な機能**
- 🗄️ **データベース**：PostgreSQL、SQLクエリ、リレーション対応
- 🔐 **認証**：メール／パスワード、Google・GitHub等のSSO
- 📁 **ストレージ**：画像・動画などのファイル保存
- ⚡ **リアルタイム**：データ変更を即座にフロントへ反映
- 🔧 **Edge Functions**：サーバーサイドのカスタム処理

---

### 1.7 Vercel × Supabase の黄金構成

**なぜ組み合わせが人気か**

```
ユーザー
  ↓
Vercel（フロントエンド / Next.js）
  ↓
Supabase（DB・認証・ストレージ）
```

| 理由 | 内容 |
|---|---|
| 開発速度 | バックエンド構築不要、その日にリリース可能 |
| 相性 | Next.jsのServer ActionsからSupabaseを直接呼べる |
| 公式統合 | Vercelダッシュボードから数クリックで接続 |
| 無料枠 | 個人開発・スタートアップ初期はほぼ無料 |
| 環境変数 | 環境変数をVercel側に登録（Key/Value形式） |

**環境変数の書き方の例**
```
NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
└──────── Key ────────┘ └────── Value ──────┘
```

**実際の採用事例**
- **Mobbin**：UIデザインリファレンスサービス（FirebaseからSupabaseへ移行）
- **Pika**：AI動画生成ツール
- 多数のYC（Y Combinator）スタートアップ

### 1.8 Supabase以前のバックエンド事情

**Supabaseが置き換えたもの（時代の流れ）**

```
【2000年代】
自前サーバー + PHP/MySQL（全部自前で構築・管理）
       ↓
【2010年代前半】
AWS（EC2 + RDS + S3）の普及（インフラは楽だが設定複雑）
       ↓
【2010年代後半】
Firebase の台頭（バックエンド不要で手軽に）
       ↓
【2020年代〜現在】
Supabase の台頭（Firebaseの使いやすさ + SQLの柔軟性）
```

**機能別：以前は何を使っていたか**

| 機能 | 以前 | 現在（Supabaseで代替） |
|---|---|---|
| DB | MySQL/PostgreSQL自前運用、AWS RDS、Heroku Postgres | Supabase Database |
| 認証 | 自前実装、Auth0、Firebase Auth | Supabase Auth |
| ストレージ | AWS S3、Cloudinary | Supabase Storage |
| リアルタイム | Socket.io（自前）、Pusher | Supabase Realtime |

**Firebase → Supabase 移行の理由**
- SQLが使いたい（FirestoreはNoSQLで複雑クエリが苦手）
- ベンダーロックイン回避（Googleへの依存を避けたい）
- 規模が大きくなった時のコスト
- オープンソース（自前サーバーへ移行可能）

### 2.2 Docker Daemonに接続できないエラー

**症状**
```
failed to connect to the docker API at unix:///var/run/docker.sock
```
`"Server": null`が出ている → Dockerデーモン（エンジン）が起動していない。

**対処法**

```bash
# 1. Docker Desktop を起動（メニューバーの🐳が緑になるまで待つ）

# 2. 動作確認
docker version

# 3. ソケットパスがない場合（macOS既知の問題）
sudo ln -sf "$HOME/.docker/run/docker.sock" /var/run/docker.sock

# 4. Docker Desktop の Settings → Advanced で
#    「Allow the default Docker socket to be used」を ON
```

**VS Code側で再試行**
- `Cmd+Shift+P` → `Dev Containers: Reopen in Container`

---

### 2.3 macOS Gatekeeper：permission denied 対応

**症状**
```bash
$ ./teams-transcription-dl
zsh: permission denied: ./teams-transcription-dl
```
macOSのGatekeeper（セキュリティ機能）が**開発元不明のアプリ**をブロックしている状態。

**解除方法**
```bash
# 検疫属性を解除
xattr -d com.apple.quarantine /path/to/file

# 実行権限を付与
chmod +x /path/to/file

# 実行
./file
```

**それでもダメなら**
1. システム設定 → プライバシーとセキュリティ
2. 「○○はブロックされました」 → 「このまま許可」
3. 再度実行

> ⚠️ **注意**：出所不明のバイナリは実行前に必ず信頼性を確認すること。

## 3. データ分析・整理

### 3.1 Excelデータ分析の必須スキル

優先度順にまとめると：

| ★ | スキル | 何ができる |
|---|---|---|
| ★★★★★ | ピボットテーブル | クロス集計・グループ化 |
| ★★★★★ | XLOOKUP / VLOOKUP | 別表からデータ参照 |
| ★★★★★ | SUMIFS / COUNTIFS | 条件付き集計 |
| ★★★★☆ | IF / IFS | 条件分岐 |
| ★★★★☆ | テーブル機能（Ctrl+T） | データの構造化 |
| ★★★☆☆ | Power Query | データ変換の自動化 |
| ★★☆☆☆ | VBA / マクロ | 定型作業の完全自動化 |

---

### 3.2 ピボットテーブルで使えるデータの条件

**ざっくり何か**
ピボットテーブルは「**縦持ち（ロング形式）のきれいな表**」を入力に取り、自由にクロス集計するツール。元データが整っていないと使えない。

**満たすべき8条件**
1. 一番上に**見出し行が1行だけ**ある
2. **1行＝1件のデータ**になっている
3. 各列は**1種類のデータ型**で揃っている
4. **結合セルがない**
5. **空白行・空白列がない**
6. **小計・合計行がデータ内に混ざっていない**
7. 空白セルはできるだけ避ける（`0`や`該当なし`を入れる）
8. **表記ゆれをなくす**（「東京」「東京都」「Tokyo」を統一）

**NG例とOK例**

```
❌ NG（クロス集計済み・横持ち）
| 部署 | 4月 | 5月 | 6月 |
| 営業 | 100 | 120 | 150 |

✅ OK（縦持ち・ロング形式）
| 部署 | 月  | 売上 |
| 営業 | 4月 | 100  |
| 営業 | 5月 | 120  |
| 営業 | 6月 | 150  |
```

**実務でのコツ**
- データは**Excelの「テーブル」（Ctrl+T）**にしておくと、行追加で範囲が自動拡大
- 整っていないデータは**Power Query**でクレンジングしてからピボットへ

## 4. ソフトウェアの構造を理解する

### 4.1 ノードとエッジ

**ざっくり何か**
**グラフ理論**から借りた概念で、複雑なシステムを「点（ノード）」と「線（エッジ）」に分解して表現する考え方。

| 概念 | 役割 |
|---|---|
| **ノード（Node、頂点）** | 「**何が**」あるか（データ・関数・モジュール・状態） |
| **エッジ（Edge、辺）** | 「**どう関係している**」か（依存・呼び出し・データの流れ） |

**ソフトウェア開発での具体例**

| 場面 | ノード | エッジ |
|---|---|---|
| 依存関係グラフ（npm等） | パッケージ | 「依存している」関係 |
| CI/CDパイプライン | タスク | 実行順序 |
| ステートマシン | 状態 | イベント・遷移 |
| AST（抽象構文木） | コードの要素 | 親子関係 |
| マイクロサービス | サービス | API呼び出し |
| ニューラルネットワーク | ニューロン | 重み付き接続 |

**派生用語**
- **DAG（Directed Acyclic Graph）**：循環のない有向グラフ。CI/CDパイプラインなど
- **ツリー構造**：1つの親しか持たない特殊なグラフ（DOM、ファイルシステム）

## 5. AI関連の主要サービス・概念

### 5.1 Ollama

**ざっくり何か**
**大規模言語モデル（LLM）を自分のPCでローカル実行**するためのオープンソースツール。クラウドAPIを使わず、モデルを手元にダウンロードして動かす。

**使用例**
```bash
# インストール後、ターミナルで
ollama run llama3.2
```
裏側でAPIサーバー（`http://localhost:11434`）が立ち上がり、他アプリと連携可能。

**メリット**
- **プライバシー**：データが外部に出ない（医療・金融・法務向き）
- **コスト**：API利用料ゼロ
- **オフライン動作**：ネット不要
- **モデル選択**：Llama、Gemma、Phi、Qwen、DeepSeekなど切替可能

**デメリット**
- 性能はPCスペック（GPU・RAM）に依存
- フロンティアモデル（Claude Opus、GPT-5）には及ばない
- 高並列の本番用途には向かない

**類似サービス**
- **LM Studio**：GUIベースで使いやすい
- **vLLM**：高並列向けの推論エンジン
- **llama.cpp**：低レベル実装、Ollamaの土台

### 5.4 Microsoft Copilot for Teams

**ざっくり何か**
**Microsoft 365 Copilot**を使って、Teamsの会議・チャット・チャネル情報を要約・活用する機能。

**取得できる3種類の情報**

1. **会議のトランスクリプト**：要約・決定事項・アクションアイテム抽出
2. **チャット**：30日間の履歴を参照
3. **チャネル投稿**：ディスカッションの要約

**設定の流れ（管理者）**

1. Teams管理センターで**文字起こし（トランスクリプト）を有効化**
2. Copilotポリシーを設定
3. Microsoft 365管理センターで**ユーザーアクセス・データアクセス**を制御
4. （任意）ユーザー辞書をアップロードして社内固有名詞の認識精度向上

**設定の流れ（会議開催者）**

会議オプションで以下から選択：
- 会議中および会議後（Copilot利用可、履歴も残る）
- 会議中のみ（履歴は残らない）
- オフ（録画・文字起こし無効）

**実用プロンプト例**
- 「これまでの会話の要点を3つに絞って」
- 「アクションアイテムを担当者・期限ごとに表にして」
- 「A案とB案それぞれの賛成・反対意見を整理して」

### 5.5 MCP（Model Context Protocol）

**ざっくり何か**
**Anthropicが2024年11月に公開**した、LLMを外部ツール・データソース・サービスに接続するためのオープン標準規格。「**AI向けのUSB-Cポート**」と例えられる。

**解決する問題**
従来、AIアプリと外部サービスをつなぐには、N×M個（AIの数×ツールの数）の個別連携が必要だった。MCPは「N+M」に削減する。

**MCPの構成要素**

| 要素 | 役割 |
|---|---|
| MCPホスト | AIアプリ本体（Claude、ChatGPT、IDE等） |
| MCPクライアント | LLMとサーバーの仲介 |
| MCPサーバー | 外部サービス側（DB、API、ファイル等） |

**代表的なMCPサーバー例**
- Notion、Slack、Gmail、Google Drive
- Cloudflare（Workers・D1・R2を操作）
- Flare（PHP/Laravelのエラー監視）
- 社内DB、社内API

**類似プロトコル**
- **OpenAPI**：API仕様の記述標準（用途は近いが、リアルタイム連携ではない）
- **Function Calling**（OpenAI等）：MCPの源流的な機能、独自実装

### 5.6 Notion MCP（実例：MCP接続のトラブルシューティング）

**ざっくり何か**
NotionのページやデータベースをClaude等のAIから直接操作できるMCPサーバー。「議事録を自動でNotionに保存」といった連携が可能。

**接続できないときに確認する3点**

**① トークンが有効か**
1. https://www.notion.so/profile/integrations にアクセス
2. Integration を選択し、「内部インテグレーションシークレット」をコピー
3. 動作確認：
```bash
curl -X GET https://api.notion.com/v1/users/me \
  -H "Authorization: Bearer secret_xxx" \
  -H "Notion-Version: 2022-06-28"
```
✅ `{"object":"user"...}` → 有効
❌ `{"code":"unauthorized"}` → 無効・要再発行

**② 対象ページがIntegrationに共有されているか**
1. Notionで対象ページを開く
2. 右上「...」→「コネクト」
3. Integration名を選んで追加（親ページ共有が子ページに必ず引き継がれるとは限らない）

**③ Claude Code側の設定が正しいか**
```bash
cat ~/.claude.json
```
で以下を確認：
- トークンが正しく貼られているか
- 余分なスペース・改行が混入していないか
- 再発行後、古いトークンのままになっていないか

### 5.7 Neuron MCP（McpConnector）

**ざっくり何か**
**PHP製のAIエージェントフレームワーク「Neuron AI」**に組み込まれている、MCP連携機能。
LaravelやSymfonyのPHPアプリから、世の中のMCPサーバーを呼び出せる。

**コード例**
```php
use NeuronAI\MCP\McpConnector;

class MyAgent extends Agent {
    protected function tools(): array {
        return [
            ...McpConnector::make([
                'url' => 'https://mcp.example.com',
                'token' => 'BEARER_TOKEN',
            ])->only(['search', 'create_issue'])  // ツール絞り込み
              ->tools(),
        ];
    }
}
```

**特徴**
- ローカル接続（command）/ リモート接続（URL）両対応
- `only()` / `exclude()`で**ツールを絞り込み可能**（誤動作防止・トークン節約）

**類似フレームワーク**
- **LangChain**（Python/JS）
- **LlamaIndex**（Python）
- **Vercel AI SDK**（TypeScript）
- **Mastra**（TypeScript）

## 6. AI駆動開発のキーワード

### 6.1 CI/CD

**ざっくり何か**
ソフトウェアの**ビルド・テスト・デプロイの流れを自動化**する手法。

| 略 | 意味 | 何をする |
|---|---|---|
| **CI** | Continuous Integration（継続的インテグレーション） | コードを統合するたびに自動でビルド・テスト |
| **CD** | Continuous Delivery / Deployment | リリース直前まで（デリバリー）／本番反映まで（デプロイメント）自動化 |

**典型的な流れ**
```
開発者がGitにpush
    ↓
CIが自動でビルド & テスト
    ↓
テスト合格でマージ可能
    ↓
CDがステージング → 本番へ自動デプロイ
```

**主なツール**
- **GitHub Actions**（最もシェアが大きい）
- **GitLab CI/CD**
- **CircleCI**
- **Jenkins**（老舗）
- **Argo CD**（Kubernetes向け）

---

### 6.2 AI駆動開発とCI/CDの関係

**重要な3つの観点**

1. **AIは速く大量にコードを書く** → CI/CDがないと検証が追いつかない
2. **AIは間違える** → CI/CDが「品質ゲート」として機能
3. **AIエージェント自身がCI/CDを使う・触る時代** → GitHub Actionsが最頻度

**ひと言でまとめると**
> AI駆動開発が「コードを書く速度」を上げ、CI/CDが「安全に届ける速度」を上げる。両方が揃って初めて、AI開発の生産性がユーザー価値に変わる。

---

### 6.3 ハーネスエンジニアリング

**ざっくり何か**
**AIエージェントが安定して正しい仕事をできるよう、モデルの「外側」の環境全体を設計するエンジニアリング手法**。
「**馬具（harness）**」が語源で、馬を制御して意図した方向へ導く比喩。

**進化の流れ**
```
プロンプトエンジニアリング（個別の指示文の工夫）
    ↓
コンテキストエンジニアリング（AIに何を見せるか）
    ↓
ハーネスエンジニアリング（環境全体の設計）
```

**4つの構成要素（OpenAIの整理）**

| 要素 | 何をする | 具体例 |
|---|---|---|
| **制約（Constrain）** | できることを制限 | レイヤー境界、依存関係ルール |
| **通知（Inform）** | 何をすべきか伝える | AGENTS.md、ドキュメント |
| **検証（Verify）** | 正しく動いたか確認 | テスト、リンタ、CI |
| **修正（Correct）** | 間違いを直させる | フィードバックループ |

**実例**
- **AGENTS.md / CLAUDE.md**：AIへの指示書を「目次」として活用（百科事典化を避ける）
- **3回ルール**：同じ違反が3回発生したらルールを1段階強化（お願い → リンタ警告 → CI強制）
- **エージェント同士のコードレビュー**：人間のレビューがボトルネックにならないように
- **リポジトリを唯一の真実のソース（SSOT）に**：Slack・口伝はAIから見えない

**有名な事例**
- OpenAIのCodexチームが**3人・5ヶ月で約100万行のコード**を出荷（人間が手書きしたコードはゼロ）

---

### 6.4 ADR（Architecture Decision Record）

**ざっくり何か**
「なぜその設計判断をしたか」を記録した小さなマークダウンファイルの集合。AI駆動開発との相性が非常によく、AIに「過去の意思決定の文脈」を渡せる。

**典型的なフォルダ構成**

```
docs/
├── REQUIREMENTS.md   # 要件（何を作るか）
├── SPEC.md           # 仕様（どう作るか）
└── adr/              # ADR（なぜそう作るか）
    ├── 001-送信方式にE案を採用.md
    └── 002-次ワークセンターの付与対象.md
```

**ADRが役に立つ理由**
- イシュー（Issue）はクローズすると見えなくなることがある
- ADRは「リポジトリ内のドキュメント」なので**AIが常に参照できる**
- 後から参加した人（AI含む）でも「なぜこの設計か」がわかる

**AI支援開発での運用ヒント**
- イシューは流動的なため**最新仕様の出所として頼らない**
- 最新状態は**ソースコードで管理**
- 仕様書は大きなマイルストーンでまとめて整備し、毎Issue単位では更新しない

---

### 6.5 Claude Codeでチーム開発を効率化するアイデア

| アイデア | 概要 |
|---|---|
| 設計仕様書の自動生成・同期 | コードからMarkdown/Wordの仕様書を自動生成 |
| GitHub PR レビューの自動化 | GitHub Actionsと組み合わせて自動レビュー |
| 要件 → 設計ドキュメントへの自動変換 | 要件書から詳細設計書の初稿を生成 |
| テスト仕様書・テストコードの自動生成 | 設計書からテスト一式を生成 |
| 管理責任者向けレビューダッシュボード | PR/Issue/Wikiを集約してSlack通知 |
| ナレッジベースの自動蓄積 | バグ修正履歴 → よくあるパターン集に変換 |
| CLAUDE.md でチームルールを標準化 | コーディング規約・ドキュメントルールを一元化 |

**CLAUDE.md の例**
```markdown
# CLAUDE.md

## コーディング規約
- MISRA-C 2012準拠
- 関数は50行以内
- 全関数にDoxygenコメント必須

## ドキュメントルール
- 設計変更時は必ずdocs/フォルダを更新
- 管理責任者承認が必要なファイル：docs/spec/

## レビュー観点
- メモリ確保は必ずNULLチェック
- 割り込み処理内でのブロッキング処理禁止
```

## 7. クラウド・インフラ

### 7.1 Cloudflare

**ざっくり何か**
**インターネットを速く・安全にする世界最大級のクラウドネットワーク**。Webサイトとユーザーの間に入って、攻撃を防ぎ、表示を速くする。

**主な機能（3本柱）**

| 機能 | 役割 |
|---|---|
| **CDN** | 世界中のサーバーから配信して高速化（CDN市場シェア約80%） |
| **WAF** | Webアプリケーション・ファイアウォール |
| **DDoS対策** | 大規模攻撃から保護（最大197Tbps） |

**その他の代表機能**
- **1.1.1.1**：無料・高速DNSサービス（プライバシー重視）
- **Workers**：エッジで動くサーバーレス環境
- **R2**：S3互換のオブジェクトストレージ
- **D1**：SQLiteベースのエッジDB
- **Zero Trust**：SASE/ゼロトラストセキュリティ
- **メールルーティング・ドメインレジストラ**

**Claudeとの連携**
公式Cloudflareコネクタ（MCP）で、自然言語でWorkers・D1・R2・KVを操作可能。

**類似サービス**
- **AWS CloudFront / WAF / Shield**
- **Akamai**
- **Fastly**

## 8. 全体マップ：用語のつながり

### AI開発の階層構造

```
  ユーザーの指示
       ↓
┌──────────────────────────┐
│ プロンプトエンジニアリング   │
│   ・指示文の工夫           │
└──────────────────────────┘
       ↓
┌──────────────────────────┐
│ コンテキストエンジニアリング  │
│   ・AIに何を見せるか        │
│   ・MCP（Notion/Slack等）  │
└──────────────────────────┘
       ↓
┌──────────────────────────┐
│ ハーネスエンジニアリング     │
│   ・環境全体の設計          │
│   ・AGENTS.md/CLAUDE.md   │
│   ・ADRで判断記録          │
│   ・CI/CDで品質ゲート       │
└──────────────────────────┘
       ↓
┌──────────────────────────┐
│ 開発インフラ               │
│   ・GitHub（コード管理）    │
│   ・CI/CD（自動化）         │
│   ・Cloudflare（公開）      │
│   ・Vercel + Supabase     │
└──────────────────────────┘
       ↓
┌──────────────────────────┐
│ AI実行環境                 │
│   ・クラウドAPI（Claude等） │
│   ・ローカル（Ollama）      │
└──────────────────────────┘
```

### Webアプリ開発の典型構成

```
ブラウザ
   ↓
Vercel／Cloudflare Pages（フロントエンド配信）
   ↓
Next.js / React（フレームワーク）
   ↓
Supabase（DB・認証・ストレージ）
   or Firebase
   or 自前バックエンド（Node.js等）
```

**この図の読み方**
- 上に行くほど「**人間の意図に近い**」、下に行くほど「**機械の実行に近い**」
- **MCP**は「コンテキスト層と外部サービスをつなぐ規格」
- **CI/CD**は「ハーネス」の検証部分を担う具体的な仕組み
- **GitHub Copilot/Claude Code**などは、これらの階層を統合した**ツール**

## 9. おすすめの学習順序

AI初心者から段階的にステップアップする順序として：

1. **基礎を固める**
   - Git/GitHub、localhost、ホスティングの概念
   - VercelとSupabaseで簡単なアプリをデプロイしてみる

2. **AIを使う**
   - GitHub CopilotまたはClaude Codeを実際に試す
   - 「補完」→「Chat」→「複数ファイル編集」と段階的にレベルアップ

3. **AIをつなぐ**
   - MCPで自分の作業環境（Notion/Slack/Gmail）と連携
   - Notion MCPで議事録自動保存などを体験

4. **AIを高速化する**
   - CI/CDを導入して自動テスト・自動デプロイを回す
   - GitHub Actionsから始めるのが最もシェアが大きく情報も豊富

5. **AIを統制する**
   - CLAUDE.md・AGENTS.mdでルール化
   - ADRで設計判断を記録
   - ハーネスエンジニアリングの考え方で環境を整備

6. **AIをホストする**
   - Ollamaでローカル実行・プライバシー対応の検討
   - エッジで動かしたいならCloudflare Workersも視野に
